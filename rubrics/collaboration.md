# Collaboration

You are evaluating the collaborative aspects of a developer's contributions. This dimension is primarily assessed at the author level rather than individual commits.

## Sub-dimensions

### Cross-module Contributions (weight: 0.30)
Does the developer contribute across different parts of the codebase?

| Score Range | Criteria |
|-------------|----------|
| 90-100 | Active contributions across 5+ distinct modules/areas |
| 70-89  | Contributions across 3-4 modules |
| 50-69  | Contributions in 2 modules |
| 30-49  | Primarily works in one area with rare exceptions |
| 0-29   | Exclusively works in a single file/directory |

### Documentation (weight: 0.25)
Does the developer maintain and update documentation?

| Score Range | Criteria |
|-------------|----------|
| 90-100 | Regularly updates docs, README, comments for complex logic |
| 70-89  | Updates docs when making significant changes |
| 50-69  | Occasionally adds documentation |
| 30-49  | Rarely documents changes |
| 0-29   | Never updates documentation, even for breaking changes |

### Mentoring Signals (weight: 0.25)
Are there signals of mentoring or collaborative development?

| Score Range | Criteria |
|-------------|----------|
| 90-100 | Co-authored commits, PR reviews referenced, pair programming signals |
| 70-89  | Occasional co-authored work, helpful commit messages for team |
| 50-69  | Standard individual contributions |
| 30-49  | Isolated work style, minimal team signals |
| 0-29   | No collaborative signals whatsoever |

### Code Review Responsiveness (weight: 0.20)
Does the commit history suggest responsiveness to feedback?

| Score Range | Criteria |
|-------------|----------|
| 90-100 | Fix-up commits addressing review feedback, iterative improvements |
| 70-89  | Some evidence of incorporating feedback |
| 50-69  | Standard commit patterns |
| 30-49  | Large monolithic commits suggesting no review process |
| 0-29   | Force-pushed history, no evidence of review |
